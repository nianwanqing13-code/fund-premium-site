## 一、这个项目是干什么的（一句话）

一个**网站**，实时计算并展示「场内基金（ETF / LOF）的溢价率」，并带三个实用功能：
1. **基金能不能申购**（申购状态）
2. **基金什么时间能申购**（申购时间提示）
3. **基金申购到账时间**（T+1 / T+2 规则）
- 另外，本次已**修复 LOF 数据缺失**问题（之前 LOF 一直是 0 只，现在能正常抓取约 370+ 只）。

溢价率 =（当日市价 − 当日预估净值 IOPV）÷ 预估净值 × 100%
- 溢价率 > 0（标红）：市价比净值贵，可能适合「卖出/赎回套利」
- 溢价率 < 0（标绿）：市价比净值便宜，可能适合「买入套利」

数据来源：[AKShare](https://github.com/akfamily/akshare)（东方财富公开行情），**免费、无需账号密钥**。

## 二、本次交接重点：四个功能 / 改动

### 功能 1：基金申购状态（能不能申购）
- **位置**：网站表格里新增的「申购状态」列。
- **显示三种状态，用颜色区分**：
  - 🟢 `开放申购`（绿色）—— 正常可买
  - 🟠 `限大额`（橙色）—— 可买但有金额上限
  - 🔴 `暂停申购`（红色）—— 当前不可买
  - 显示 `-` 表示数据源没返回（空值）
- **怎么实现的**：
  - 后端 `app.py` 里的 `get_subscribe_map()` 函数，调用 `ak.fund_open_fund_info_em()`（AKShare 的东方财富接口）拿到每只基金的原始申赎状态。
  - 用 `_normalize_subscribe_status()` 把原始中文标准化成上面三种状态，并过滤掉空值/NaN。
  - 数据带 10 分钟缓存（避免频繁请求被限）。
- **参考依据**：GitHub 上有类似项目 `hkt321/arbitrage-alert`，它也是用东方财富取申赎状态，证明这个数据源是对的。

### 功能 2：基金申购时间（什么时间能申购）
- **位置**：网站顶部的一个提示条，实时显示「当前可申购 / 暂不可申购」。
- **怎么实现的**：
  - 申购时间**没有现成接口可查**，是固定规则：交易日（周一至周五，非法定节假日）的 **9:30–11:30、13:00–15:00** 才能申购。
  - 前端 `templates/index.html` 按当前时间判断并提示。
  - GitHub 上没找到专门做「申购时间」的项目，所以这套固定规则是合理做法。

### 功能 3：基金申购到账时间（新增列）
- **位置**：表格里「申购状态」右侧新增的「到账时间」列。
- **显示规则**（前端 `arriveText()` 函数，按类型 + 申购状态判断）：
  - **ETF** → `T+1`（场内 ETF 多为实物申赎，约次日到账可卖）
  - **LOF** → `T+2`（场内 LOF 申购份额约两个交易日后到账可卖）
  - 若该基金 `暂停申购` → 显示 `—`（不适用）
- **说明**：这是通用惯例规则，具体以各基金公司公告为准；页脚已注明。

### 功能 4：修复 LOF 数据缺失（重要）
- **背景**：交接时表格里的 LOF 一直是 **0 只**——原代码用 `ak.fund_lof_spot_em()`，但东方财富该接口会**连接重置（RemoteDisconnected）**，且返回字段里**没有 IOPV**，导致 LOF 全部被跳过。
- **修复方式**：在 `app.py` 用自定义 `fetch_lof_list()` 替换它（见第五节「关键技术改动」）。
- **现状**：LOF 现在能正常抓取（约 **370+ 只**），与 ETF 合计约 **1890 只**场内基金。

## 三（补充）、关键技术改动（接手人务必看，别把修复改回去）

### 1. 数据抓取不再依赖 akshare 的 `fund_etf_spot_em` / `fund_lof_spot_em`
这两个接口有两个坑，已绕过：
- **连接被重置**：东方财富对缺请求头的连接会 `RemoteDisconnected`；现统一带 `User-Agent` / `Referer` 请求头。
- **分页坑（最致命）**：东方财富 `clist` 接口**默认每页 100 条，传超大 `pz` 会被静默钳回 100**。
  原代码单次请求只拿到 100 行，导致 ETF 只有 90 只、LOF 只有 81 只（全量应是 1500+ / 370+）。
  **正确做法**：先请求第一页拿到 `data.total`，再按页（`pn=2,3,...`）翻完——代码里 `_fetch_eastmoney_clist()` 已实现。

现在 ETF / LOF 都走 `app.py` 里的：
- `fetch_etf_list()`：字段含 `f441`（IOPV 实时估值），溢价率 = (市价 − IOPV) / IOPV。
- `fetch_lof_list()`：字段含 `f402`（基金折价率）。**LOF 没有 IOPV**，溢价率用折价率反推：`溢价率 = −折价率`。
- 共同底层：`_fetch_eastmoney_clist(fs, fields, page_size=1000)`，带 4 次重试 + 分页。

### 2. 性能优化（首屏从 ~40s 降到 <1s）
- 去掉了 akshare 自带的 **0.5~1.5 秒随机延迟**（翻页时只留 0.1s 间隔）。
- 加了 `_fetch_lock` 抓取锁：避免并发请求各自触发一次完整抓取互相拖慢。
- **启动预热缓存**：`app.py` 在 `__main__` 里先调一次 `get_fund_premium()` 把数据抓好再对外服务，首个访问直接命中缓存（约 90ms 返回）。

### 3. 缓存与超时
- ETF / LOF 行情缓存 60 秒（`_CACHE_SECONDS`）。
- 申购状态缓存 10 分钟。
- 网页每 60 秒自动刷新，也可点「刷新数据」立即拉取。

## 七、常见问题

**Q：页面一直"加载中"或报错？**
A：多半是开了代理/VPN。关掉系统代理后刷新。数据源需要直连。

**Q：申购状态列很多显示 `-`？**
A：那是数据源（东方财富）没返回该基金的申赎状态，属正常，不是 bug。

**Q：公网链接（trycloudflare）打不开？**
A：那是临时隧道，每次重启会变，且必须保持本机开机、网站运行、代理关闭。想要固定域名需注册 Cloudflare 配固定隧道。

**Q：推送到 GitHub 时弹出要输用户名密码？**
A：正常情况下 `推送更新.bat` 会弹浏览器授权。若真要填，用 GitHub 用户名 + 一个**新生成的 PAT**（GitHub → Settings → Developer settings → Personal access tokens，勾 `repo`）当密码填。

**Q：LOF 数量很少或为 0？**
A：早期用 `ak.fund_lof_spot_em()` 会发生连接重置且缺 IOPV，已改成自定义 `fetch_lof_list()`（带请求头 + 分页 + 重试）。若仍异常，查 `server_err.log` 看是否有「获取基金列表失败」；多半是东方财富临时限流，等 1 分钟刷新即可（行情有 60s 缓存）。非交易时间部分基金不返回行情也属正常。

**Q：溢价率、到账时间怎么算的？**
A：ETF 溢价率 = (市价 − IOPV) / IOPV，IOPV 来自字段 `f441`；LOF 没有 IOPV，用「折价率 f402」反推，溢价率 = −折价率（折价率为负即溢价）。到账时间：ETF→T+1、LOF→T+2、暂停申购→「—」（通用规则，以基金公司公告为准）。
